# 對話流程

我們的做法不是讓 AI「回答」問題，而是讓 AI「翻譯」問題。

使用者問「本月營收多少？」，AI 的工作是把這句話翻譯成 SQL：`SELECT SUM(total_amount) FROM orders WHERE order_date >= '2026-04-01' AND status NOT IN ('cancelled', 'refunded')`。系統拿這句 SQL 去真實資料庫查，資料庫回傳 132619，格式化成 NT$132,619 給使用者看。

**數字來自資料庫，不是 AI 編的。** AI 只負責「聽懂問題 → 寫出正確的 SQL」，準不準取決於 SQL 寫得對不對，不是 AI 記了什麼。

## 跟純 LLM 差在哪？

| | 純 LLM（如直接問 ChatGPT） | 我們的做法（Text-to-SQL） |
|---|---|---|
| 資料來源 | AI 訓練時記住的（可能過時） | 真實資料庫（即時） |
| 會編數字嗎 | 會（幻覺） | 不會，數字來自 DB |
| 能查你公司的資料嗎 | 不能 | 能，每個客戶有自己的 DB |
| AI 的角色 | 知識來源 | **翻譯官**（中文 → SQL） |

## 跟 RAG 差在哪？

| | RAG | 我們的做法 |
|---|---|---|
| 流程 | 問題 → 搜相關文件 → AI 摘要回答 | 問題 → AI 寫 SQL → 資料庫算 |
| 適合什麼 | 文字類問題（政策、知識庫） | **數字類問題**（營收、數量、排行） |
| 精準度 | 摘要可能漏掉或曲解 | **精確到元**，因為是資料庫直接算 |
| 會算錯嗎 | 可能，AI 做數學不可靠 | 不會，SUM/COUNT 是資料庫算的 |

**一句話總結：** RAG 是讓 AI 讀文件後回答，我們是讓 AI 寫 SQL 後讓資料庫回答。數字的事交給資料庫算，不交給 AI 算。

## 使用者問題要很精確嗎？

不用，沒有想像中嚴格。

**AI 聽得懂的原因：**

1. **Schema 有中文名稱** — AI 看到 `total_amount = 訂單金額`、`order_date = 訂單日期`，所以使用者說「營收」它知道要找 `total_amount`
2. **對話歷史** — 問完「本月營收」追問「上個月呢」，AI 看得到前面在聊什麼
3. **產業別提示** — 告訴 AI 這是餐飲業，「營收」和「毛利」在不同產業定義不同

**這些問法都能懂：**
- 「本月營收多少」→ 正常
- 「這個月賺了多少錢」→ 也行，AI 知道「賺錢」≈ 營收
- 「revenue this month」→ 也行，多語言
- 「四月訂單」→ 也行，知道是 2026 年 4 月

**真正會出問題的是：**
- 「那個東西多少」→ 太模糊，AI 不知道「那個」是什麼
- 「幫我算 A 客戶扣掉退貨後的淨額再乘以稅率」→ 太複雜，一句 SQL 很難寫對

這就是為什麼有**信心度機制** — AI 自己不確定時不會硬答，會反問使用者講清楚一點。

## 使用者問一個問題後，系統做了什麼？

1. **送出訊息** — 畫面顯示使用者的問題氣泡，同時送到後端處理
2. **找到這個客戶的資料庫** — 每個客戶有自己的 DB，系統知道這個人屬於哪間公司
3. **讀資料庫結構** — 讀取這間公司的所有表格和欄位的中文名稱（例如 `orders 表 = 訂單`、`total_amount = 訂單金額`）。薪資、銀行帳號等敏感欄位直接排除，AI 看不到
4. **問 AI** — 把資料庫結構、之前的對話紀錄（最多 10 輪）、統計口徑規則（排除取消/退款、應計基礎）、使用者的問題一起送給 AI。AI 回傳一句 SQL、一句中文回覆模板、它覺得自己多有把握（0-100%）
5. **安全檢查** — 只允許查詢，不允許刪除或修改資料。再次確認沒碰到敏感欄位。沒通過就告訴使用者「此資料受限，無法查詢」
6. **判斷 AI 有多少把握** — 高（95% 以上）直接查資料庫回傳結果。中（70-95%）查資料庫但附註「建議確認」+ 可以看 SQL。低（70% 以下）不查，請使用者把問題說得更清楚
7. **查資料庫** — 用 AI 產生的 SQL 去這間公司的 DB 撈資料，只查到今天，不含未來日期
8. **格式化結果** — 數字四捨五入無小數（NT$1,234,567），表格加上欄位名稱
9. **存對話紀錄** — 使用者說了什麼 + AI 回了什麼，存到資料庫，下次可以回看
10. **畫面顯示結果** — 數字用綠色大字、表格用有標題的表格、不確定用文字引導、錯誤用紅色訊息。AI 回覆用 markdown 渲染（標題、列表、粗體等）。同時更新左邊的對話清單

## 多輪對話怎麼運作？

使用者問完「本月營收多少」後追問「跟上個月比呢」，系統會把前面的對話（最多 10 輪）一起送給 AI。AI 看到之前在聊營收，知道「上個月」是指上個月的營收，產生正確的 SQL，不需要使用者重複說明。

## 敏感資料怎麼保護？

兩道防線。第一道：組 AI 提示時，薪資、銀行帳號等欄位根本不給 AI 看，AI 不知道有這些欄位，自然不會查。第二道：萬一 AI 還是產出了碰到敏感欄位的 SQL，系統攔截不執行，回覆「此資料受限」。

## Dashboard 和 Chat 怎麼對齊數字？

兩邊用同一套統計口徑。排除取消和退款（`status NOT IN ('cancelled', 'refunded')`）。pending 算營收（應計基礎，訂單成立就算）。只算到今天，不含未來日期的資料。AI 的 system prompt 有寫明這些規則，引導 LLM 產出的 SQL 用相同的過濾條件。

## 寫入操作（待開發）

未來使用者說「幫我新增一筆訂單」時，AI 產生 INSERT SQL 後不會直接執行，而是先顯示操作摘要讓使用者確認（例如「即將新增：客戶 ABC、金額 NT$12,500」）。使用者按確認才執行，按取消就丟棄。所有寫入都要使用者親手按確認，不會自動執行。
