## 問題陳述

ERP 開發公司（即本團隊）每次接案都從零幫客戶建 ERP 系統：設計 schema、寫程式、建 UI。每個客戶需求不同，導致大量重複但又略有差異的人工開發工作。客戶端則面臨兩個痛點：建系統貴又慢、建好後操作複雜老闆沒時間學。

傳統 ERP 的表單、選單、報表 UI 是人類操作資料庫的必要中介。AI 帶來結構性改變的機會：用聊天介面取代整個傳統 UI 層，讓使用者用對話完成所有 ERP 操作（查詢、新增、修改、刪除），團隊只需專注在 schema 設計 — 這本來就是團隊最強的部分。

## 需求證據

- 團隊本身就是 ERP 開發公司，每天在接客戶的開發需求，親身經歷這個痛點
- 客戶反覆提出類似需求（進銷存、報表、查詢），代表 pattern 明確可抽象化
- 老闆（公司決策者）主動提出要做這個產品，代表內部已有共識
- 系統交付後客戶老闆通常不自己進系統操作 — 聊天介面直接解決這個問題

## 現狀

- 客戶付開發費，團隊從零寫 code（schema + 程式 + UI）
- 每個案子都有重複的部分（CRUD、報表、權限），但因需求差異無法完全複用
- 系統交付後，客戶操作學習成本高，老闆通常不自己進系統查資料
- 傳統 ERP 的 UI 開發佔了交付時間的大部分 — 表單、列表、報表都是圍繞同一份資料的不同呈現方式

## 產品定義：AI ERP

**核心概念：** 砍掉傳統 ERP 的整個 UI 層（表單、選單、報表），用一個能讀能寫的 AI 聊天介面 + Dashboard 取代。團隊為客戶設計 schema 並部署，同一套聊天介面接上任何 schema 就是一套完整的 ERP。

**目標客戶：** 需要 ERP 系統的中小企業（現有接案客戶為第一批）

**產品組成：**

| 元件 | 功能 | 取代什麼 |
|------|------|----------|
| AI 聊天介面 | 自然語言完成所有 CRUD 操作 + 資料查詢 | 傳統 ERP 的表單、列表、選單 |
| Dashboard | 業務概覽（關鍵指標、即時狀態） | 傳統 ERP 的報表頁面 |
| 檔案上傳 | 批量資料匯入（Excel/CSV） | 手動逐筆輸入 |
| Schema 服務 | 團隊手動為客戶設計並部署資料庫 schema | 不變，這是團隊核心專業。未來可考慮 AI 輔助 schema 設計作為內部工具 |

**初始版本不包含：** 傳統 CRUD 表單介面、Chat-to-build（AI 自動產生 schema）、free-form UI 生成。未來如有需要可按需加入。

## 限制條件

- 查詢準確率分級要求（以 Golden Test Suite 驗證）：關鍵財務數據（營收、應收帳款）> 99%，一般查詢（庫存查詢、人員列表）> 95%。商業數據報錯數字信任直接歸零。生產環境中未見過的查詢類型，關鍵財務一律附帶 SQL 供使用者確認
- 寫入操作一律需使用者確認後才執行，不允許靜默寫入
- 台灣中小企業市場為初始目標，需支援在地商業邏輯（統一發票、營業稅、勞健保）
- 團隊有 PHP/Laravel 技術背景，現有客戶系統為 PHP + MySQL
- 需考慮多租戶架構（每個客戶一套獨立系統）
- 操作回應延遲：首次回應（first token）< 3 秒，完整回應 < 10 秒（配合 SSE 串流架構，使用者在 3 秒內看到回應開始）
- 需符合台灣個人資料保護法，客戶商業數據嚴格隔離

## 安全性與合規

- **認證：** Laravel Sanctum（API token）。OAuth2 + JWT 規劃於未來開放第三方整合時再導入
- **資料隔離：** 多租戶間嚴格隔離，任何操作只能存取該租戶的資料
- **加密：** 資料庫連線 TLS，敏感欄位（薪資、成本）加密儲存
- **個資法合規：** 符合台灣個人資料保護法，提供資料匯出/刪除功能
- **寫入安全：** 見下方「寫入操作安全模型」章節

### 寫入操作安全模型

聊天介面支援完整 CRUD（SELECT / INSERT / UPDATE / DELETE），寫入操作的安全機制如下：

**一律確認原則：** 所有寫入操作在執行前必須顯示操作摘要，取得使用者明確確認。

**確認流程：**
1. 使用者下達指令（如「幫我新增一筆訂單，客戶 ABC，金額 12,500」）
2. AI 產生對應的 SQL mutation 並回傳人類可讀的操作摘要（「即將新增 1 筆訂單：客戶 ABC，金額 $12,500。確認？」）
3. 操作以 `pending` 狀態暫存
4. 使用者確認後才執行；拒絕則丟棄
5. 執行結果回饋使用者（「已新增訂單 #1234」）

**寫入錯誤處理：**
- 確認後執行失敗（如違反 unique 約束）：回報具體錯誤原因，不做部分寫入，整筆 rollback
- 批量匯入部分失敗：以 transaction 包裹，全成功或全失敗，回報失敗行數和原因
- Audit trail 寫入失敗：操作本身仍執行但記錄告警，由系統通知團隊補登

**SQL 安全防線：**
- 禁止 DDL 操作（DROP、ALTER、TRUNCATE）
- UPDATE / DELETE 必須帶 WHERE 子句，禁止全表操作
- 批量操作設定影響行數上限，超過需額外確認
- 所有寫入操作記錄完整 audit trail（操作者、時間、SQL、影響行數）

**核心原則：寧可多確認一次，也不執行錯誤的操作。**

### 查詢操作安全模型（信心度分層）

- **高信心（> 95%）：** 直接回答，背景記錄 SQL 和結果供審計
- **中信心（70-95%）：** 回答但附提示「此查詢較複雜，建議確認」，可展開看 SQL
- **低信心（< 70%）：** 不回答，改為引導使用者釐清意圖（「你是要查 A 還是 B？」）

核心原則：寧可不回答，也不回答錯的數字。

## 方案

團隊為客戶設計 schema 並部署 AI ERP 平台，客戶透過聊天介面完成所有 ERP 操作。過去接案累積的 pattern 作為 domain knowledge，幫助 AI 更準確地理解各產業的 schema 結構和商業邏輯。

每個客戶的交付從「schema + 程式 + UI」簡化為「schema + 同一套聊天介面」，大幅縮短交付時間和降低開發成本。

### 功能範圍

後端回傳 JSON、前端透過 Blade + Alpine.js + Axios 呼叫。元件庫定義見 [00-component-library.md](../spec/00-component-library.md)。

**聊天操作引擎：**
- 自然語言轉 SQL（SELECT / INSERT / UPDATE / DELETE）
- 寫入操作確認流程（pending → confirm/reject → execute）
- 信心度分層回退（查詢操作）
- 對話歷史與多輪對話
- 敏感欄位保護
- 完整 audit trail
- 品質閘門：Golden Test Suite，查詢 150 筆（已完成）+ 寫入操作測試（待建立）

**Dashboard：**
- 業務關鍵指標概覽（營收、訂單數、庫存狀態等，依客戶 schema 動態產生）
- 即時狀態卡片
- 依 schema metadata 自動識別可呈現的指標：`schema_metadata` 表中標記 `is_kpi: true` 的欄位 + 金額/數量型欄位自動彙總
- Dashboard 的指標查詢使用預定義 SQL（非即時 LLM 生成），確保穩定性和速度

**檔案上傳：**
- 支援 Excel（.xlsx）/ CSV 格式，編碼支援 UTF-8 和 Big5（台灣常見）
- 檔案大小上限 10MB（約 10 萬筆資料）
- 上傳後 AI 解析欄位並顯示預覽（「此檔案包含 47 筆資料，欄位：品名、數量、單價。匯入到 products 表？」）
- 欄位名稱不完全匹配時，AI 建議對應（如「商品名稱」→ `product_name`），使用者確認
- 經使用者確認後批量執行（transaction 包裹，全成功或全失敗）
- 重複資料偵測：依據主鍵或 unique 欄位提示「第 3、15 行可能重複」
- 執行結果摘要回饋

**認證與租戶隔離：**
- 登入 / 忘記密碼 / 重設密碼
- DB-per-tenant 多租戶隔離
- API token 認證（Sanctum）

**前端頁面：**
- 登入 / 忘記密碼 / 重設密碼頁
- 聊天主頁（含 Dashboard 概覽）
- 首次使用引導（AI 主動介紹系統能做什麼）

後端 spec 詳見 [01-phase1-backend.md](../spec/01-phase1-backend.md)，前端 spec 詳見 [02-phase1-frontend.md](../spec/02-phase1-frontend.md)。

### 技術方向

| 層級 | 技術選擇 | 說明 |
|------|----------|------|
| Backend | PHP + Laravel | 團隊現有技術棧 |
| Database | MySQL | 團隊熟悉，現有客戶系統一致 |
| AI Layer | Apertis（OpenAI-compatible）+ gpt-4.1-mini + function calling | Text-to-SQL 準確率高，支援 function calling |
| Frontend | Blade + Alpine.js + Axios | Laravel 原生模板，Alpine.js 管理聊天狀態，Axios 處理 API 請求 |
| UI 設計 | Claude DESIGN.md | 溫暖聊天介面風格，適合非技術用戶 |
| 多租戶 | DB-per-tenant | 每個客戶獨立 MySQL DB |

### 多租戶策略

採用 DB-per-tenant：每個客戶獨立 MySQL DB，完全隔離、獨立備份/還原。每個客戶的 schema 由團隊客製設計，獨立 DB 確保租戶間完全隔離。

### 領域知識層定義

團隊過去接案累積的產業知識，以結構化形式儲存供 AI runtime 使用：

- **形式：** 結構化 JSON/YAML schema library，按產業分類（餐飲、製造、貿易...）
- **內容：** 各產業常見的 table 定義、欄位命名慣例、關聯關係、商業規則（如：營業稅 5%、發票格式）
- **使用方式：** 作為 LLM 的 system prompt context，AI 參考 schema metadata 提高 SQL 準確率（查詢和寫入操作皆適用）
- **成長機制：** 每個新客戶的系統經人工審核後回饋到 library

### 技術風險與應對

| 風險 | 嚴重度 | 應對策略 |
|------|--------|------------|
| Text-to-SQL 查詢準確率不足 | 高 | 用 function calling 取代 raw SQL 生成；預定義常見查詢 template；SQL 執行前經 EXPLAIN 驗證；關鍵財務查詢加信心度分層 |
| 寫入操作產生錯誤 SQL | 高 | 一律確認機制兜底；SQL 安全防線（禁止 DDL、必須帶 WHERE）；完整 audit trail 可追溯；影響行數上限控制 |
| LLM 成本失控 | 中 | 初步估算：每次操作 ~2K tokens（$0.01-0.03/次），10 客戶 x 20 次/天 = $60-180/月。定價需覆蓋 LLM 成本 + 30% margin。考慮 cache 常見查詢、使用較小模型處理簡單操作 |
| 多租戶資料洩漏 | 高 | DB-per-tenant 物理隔離；API 層 tenant ID 強制校驗 |
| 純聊天介面無法覆蓋特定 ERP 場景 | 中 | 以純聊天 + 檔案上傳為起點，實際使用中若發現特定場景不適用再按需補充介面。Dashboard 提供視覺化概覽降低純文字的侷限 |

## 成功標準

- [ ] 至少 1 個現有客戶使用 AI ERP，且日常使用（每週 > 5 次操作，含查詢和寫入）
- [ ] 查詢準確率：關鍵財務數據 > 99%，一般查詢 > 95%（以 Golden Test Suite 驗證）
- [ ] 寫入操作準確率：AI 產生的操作摘要與使用者意圖一致率 > 90%（以寫入 Golden Test Suite 驗證）
- [ ] 客戶能透過聊天 + 檔案上傳完成日常 ERP 操作，無需傳統表單介面

## 商業模式

- **現階段（服務模式）：** 團隊為客戶設計 schema、部署 AI ERP 平台、收取服務費。每個客戶的交付時間因砍掉 UI 開發而大幅縮短
- **未來（SaaS 轉型）：** 產品成熟後考慮自助註冊、訂閱付費、線上金流。同一套聊天介面可服務任何 schema，SaaS 化的邊際成本極低

- 寫入操作的 LLM 準確率（需透過 Golden Test Suite 驗證）
